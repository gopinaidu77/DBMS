Unit – III: ER Modeling and Normalization [cite: 62]
Short Answer Questions (4 lines each) [cite: 62]
1. Define Entity and Entity Set. [cite: 63]

An entity is a real-world object. An entity set is a group of similar entities. [cite: 63]

Example: A student is an entity, and all students form an entity set. [cite: 64]

2. What is a Relationship Type? [cite: 65]

It represents an association between entities. Example: A student 'enrolls' in a course – ‘enrolls’ is the relationship type. [cite: 65]

3. Differentiate between Strong and Weak Entity. [cite: 66]

A strong entity has a primary key. [cite: 66]

A weak entity cannot exist without a strong entity and uses a foreign key. [cite: 67]

4. Define Key and Attribute in ER Model. [cite: 68]

Key uniquely identifies an entity. [cite: 68]

Attributes describe properties of an entity like name, age, etc. [cite: 69]

5. What are Structural Constraints? [cite: 70]

They define cardinality (1:1, 1:N, N:M) and participation (total or partial) in relationships. [cite: 70]

6. Define Functional Dependency. [cite: 71]

A functional dependency X → Y means if two tuples agree on X, they must agree on Y. It’s used in normalization. [cite: 71]

7. What is Normalization? [cite: 72]

Normalization is a process to reduce data redundancy and improve data integrity by organizing data into tables. [cite: 72]

8. What is 3NF? [cite: 73]

A relation is in 3NF if it is in 2NF and no transitive dependency exists between non-prime attributes. [cite: 73]

Long Answer Questions (8+ lines each) [cite: 74]
1. Explain different types of attributes in ER Model. [cite: 74]

Attributes can be: [cite: 75]

• Simple (indivisible) [cite: 75]

• Composite (divided into sub-parts) [cite: 75]

• Derived (calculated) [cite: 75]

• Multi-valued (multiple values) [cite: 75]

They help in fully defining an entity’s properties and relationships. [cite: 75]

2. What are ER diagrams? Describe symbols used. [cite: 76]

ER diagrams represent entities, relationships, and attributes. [cite: 76]

Rectangles = entities, Diamonds = relationships, Ovals = attributes. Lines connect elements. Double lines = total participation. [cite: 77]

This diagram helps in visualizing DB design. [cite: 78]

3. Discuss Functional Dependencies with examples. [cite: 78]

A → B implies B is functionally dependent on A. Example: Student_ID → Student_Name. [cite: 79]

Functional dependencies help detect redundancy and guide normalization. [cite: 80]

4. Differentiate between Partial and Transitive Dependency. [cite: 80]

Partial Dependency: Non-key depends on part of composite key (1NF to 2NF). [cite: 81]

Transitive Dependency: A → B and B → C implies A → C (2NF to 3NF). [cite: 82]

These need to be removed for normalized DB design. [cite: 83]

5. Explain the purpose and process of normalization. [cite: 83]

Normalization reduces redundancy and anomalies. It involves organizing data into normal forms: 1NF (atomic), 2NF (no partial), 3NF (no transitive), BCNF (more strict). [cite: 84]

Each form improves structure and integrity. [cite: 85]

6. What is the difference between 1NF, 2NF, and 3NF? [cite: 85]

• 1NF: Atomic values [cite: 86]

• 2NF: No partial dependencies [cite: 86]

• 3NF: No transitive dependencies [cite: 86]

Each step removes redundancy and improves clarity of data storage. [cite: 86]

7. Explain problems like Fan Traps and Chasm Traps in ER modeling. [cite: 87]

Fan Trap occurs when a single entity is related to multiple entities without proper linkage. [cite: 88]

Chasm Trap happens when a relationship is missing, preventing data retrieval. These are resolved using correct relationship mapping. [cite: 89]

8. Describe the Database Design Methodology using ER to Relational. [cite: 90]

Steps: [cite: 91]

• Identify entities and relationships [cite: 91]

• Define primary and foreign keys [cite: 91]

• Convert attributes to columns [cite: 91]

• Normalize tables [cite: 91]

This method ensures a logical, efficient relational schema. [cite: 91]

Unit – IV: Transaction Management and Concurrency Control [cite: 92]
Short Answer Questions (4 lines each) [cite: 92]
1. What is a Transaction? [cite: 93]

A transaction is a sequence of operations performed as a single logical unit of work. [cite: 93]

It must follow ACID properties to ensure database consistency. [cite: 94]

2. Define ACID Properties. [cite: 95]

ACID stands for Atomicity, Consistency, Isolation, and Durability. These ensure safe and reliable transactions in DBMS. [cite: 95]

3. What is Atomicity? [cite: 96]

Atomicity ensures that all operations in a transaction are completed; [cite: 96]

if not, the transaction is aborted and changes are rolled back. [cite: 97]

4. What is a Schedule? [cite: 98]

A schedule is an ordered list of operations from a set of transactions. [cite: 98]

It defines the execution order in a multi-user environment. [cite: 99]

5. What is Serializability? [cite: 100]

Serializability ensures that a schedule results in the same database state as some serial execution of transactions. [cite: 100]

6. Define Deadlock. [cite: 101]

Deadlock occurs when two or more transactions wait indefinitely for each other to release resources. [cite: 101]

7. What is Two-Phase Locking Protocol? [cite: 102]

It is a concurrency control method that divides a transaction into two phases: growing (acquire locks) and shrinking (release locks). [cite: 102]

8. What is Concurrency Control? [cite: 103]

Concurrency control manages simultaneous execution of transactions to prevent conflicts and ensure consistency. [cite: 103]

Long Answer Questions (8+ lines each) [cite: 104]
1. Explain ACID properties with examples. [cite: 104]

• Atomicity: All steps must succeed or fail as a whole. Example: Money transfer. [cite: 105]

• Consistency: Ensures database remains valid after transaction. [cite: 106]

• Isolation: Concurrent transactions do not interfere. [cite: 106]

• Durability: Once committed, changes are permanent. These properties protect database integrity during crashes and errors. [cite: 107]

2. Describe different states of a transaction. [cite: 108]

Transaction states include: [cite: 108]

• Active: Performing operations [cite: 108]

• Partially Committed: Last statement executed [cite: 108]

• Committed: Changes saved [cite: 108]

• Failed: Error occurred [cite: 108]

• Aborted: Rolled back These states help in managing transaction lifecycle effectively. [cite: 108]

3. What is Serializability? Explain types. [cite: 109]

Serializability ensures consistency in concurrent transaction execution. Types: [cite: 109]

• Conflict Serializability: Based on non-conflicting operations. [cite: 109]

• View Serializability: Based on data views. Both types aim to ensure correctness without necessarily using serial order. [cite: 110]

4. Discuss Concurrency Control Techniques. [cite: 111]

• Lock-based Protocols: Use shared/exclusive locks. [cite: 111]

• Timestamp-based Protocols: Use timestamps for ordering. [cite: 111]

• Optimistic Concurrency Control: Assumes low conflict, checks at commit. Each technique balances performance and consistency. [cite: 112]

5. Explain Two-Phase Locking Protocol with example. [cite: 113]

It has two phases: [cite: 113]

• Growing phase: Transaction acquires all needed locks. [cite: 113]

• Shrinking phase: Locks are released and no new locks can be acquired. [cite: 114]

Example: T1 locks A and B, performs operations, then releases. Ensures serializability but may lead to deadlocks. [cite: 115]

6. Describe Deadlock and its prevention methods. [cite: 116]

Deadlock occurs when transactions hold and wait on resources. [cite: 116]

Prevention methods: [cite: 117]

• Timeouts: Abort after wait time [cite: 117]

• Wait-Die / Wound-Wait: Based on priority [cite: 117]

• Deadlock Detection and Recovery: Periodic checks and rollback Proper handling ensures system availability. [cite: 117]

7. Compare Pessimistic and Optimistic Concurrency Control. [cite: 118]

• Pessimistic: Uses locking, assumes conflict is likely. [cite: 118]

• Optimistic: Assumes low conflict, checks at end. Pessimistic is safer but slower. Optimistic is faster but needs rollback mechanism. [cite: 119]

Choice depends on workload nature. [cite: 120]

8. What is Transaction Logging and Checkpointing? [cite: 121]

• Logging: Records all transaction activities to a log file. [cite: 121]

• Checkpointing: Saves current state to minimize recovery time. [cite: 121]

These help in recovering the database to a consistent state after a crash. [cite: 122]
